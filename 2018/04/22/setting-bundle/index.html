Setting.bundle 的简单使用

发表于 2018-04-22 |

简介

作为iOS的开发者可能需要打各种环境的包:比如dev、staging、production等。本文基于Settings.bundle 配置APP的Host Domain.

效果如下:

项目需求

在开发中,我们常用到的环境就是debug和release.但是我们希望配置域名地址只在我们开发测试阶段使用,上线包是不包含此功能的。

所以我们可以需要单独构建一份新的release环境改名为staging用于打包测试。

除此外,为了方便后端同学有的时候可能需要连接本地的服务器调试,还需要提供一个可以自行编辑的域名或请求地址。

需求确定后,接下来着手准备。

配置Setting.bundle

创建Setting.bundle

首先,根据项目的需求,我们需要新建两种Setting.bundle,一种用于我们平时模拟器的开发使用Debug,一种用于打包给测试同学测试staging。

创建相关目录

在工程目录下创建文件夹Setting以及子文件夹Debug和Staging(一般生产环境不需要setting, 当然这要依据具体的业务需求)

添加Setting.bundle

选择上面创建的文件路径,分别添加Setting.bundle。

新建Setting.bundle,记得不要勾选target.

创建配置

点击工程文件, 选中info, 在Configurations下点击+, 复制一个Configuration(我一般复制Release, 这样发出去的ipa都去掉日志等不必要的操作 具体依据业务需求修改)

重命名

添加拷贝脚本

因为上面我们创建的时候,没有添加target。所以项目中找不到Setting.Bundle。

我们需要用脚本把Setting.Bundle拷贝到ipa中.
选中”Target“, 切换到Build Phases, 点击+, 选中New Run Script Phase。

修改Setting.bundle

根据上面的项目需求,我们需要在Preference Items中创建两个item,类型分别为Multi Value和Text Field。

  • Multi Value:
    为我们提供内置多种的域名地址
  • Text Field
    为我们提供自定义域名地址的功能

项目踩坑

由于手上的项目存在cocos2d的target工程，而且项目使用了cocoapods管理第三方库和公司的私有组件库。所以在调试的时候还是遇到了一些问题,在此记录一下。

因为我们配置了staging环境,在build生成libpod.a文件时会找不到,所以我们尽量将staging环境下配置等同于release环境中的配置。

  • 选择pod环境

CONFIGURATION_BUILD_DIR

  • Build Products Path

  • LIBRARY_SEARCH_PATHS

  • PODS_CONFIGURATION_BUILD_DIR

CONFIGURATION_TEMP_DIR

如果pod中CONFIGURATION多了staging,可移除。
